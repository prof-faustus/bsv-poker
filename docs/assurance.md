# Software assurance & security practice

This build adopts a documented assurance practice (REQ-ENG-001, REQ-APP-170) and a security practice
(REQ-APP-172). Neither is a compliance claim; both are the concrete, enforced engineering discipline
the repository runs in CI.

## Assurance practice (NPR 7150.2, adapted)

- **Requirements register.** Every `REQ-*` is a machine-readable entry (`spec/requirements.yaml`,
  generated by `tools/extract-requirements.ts`; REQ-ENG-002).
- **Traceability gate.** `tools/trace-map.ts` + `tools/traceability.ts` map each satisfied requirement
  → source → passing test; CI fails on an untraced consensus file (REQ-ENG-003). Current: 144/223.
- **Reproducibility.** `tools/reproduce.ts` regenerates every committed vector and exits non-zero on
  any mismatch (REQ-TEST-005).
- **Test taxonomy** (REQ-APP-160): unit → property → **interpreter-level script** (negatives fail
  *inside* the real interpreter) → E2E → adversarial (`packages/sdk/test/adversarial.test.ts`) →
  on-chain (`tools/onchain-*-e2e.ts`). 181 TS + 16 Go tests run in `node tools/ci.ts`.
- **Power-of-Ten adaptation (not literal; REQ-ENG-001).** Documented in
  `docs/adr/0003-self-contained-interpreter.md`: rules requiring static allocation / no-recursion /
  pointer limits do not apply literally in a GC'd TS/Go runtime. The *measurable* rule — bounded
  working memory in the hot path — is asserted by `tools/perf-suite.ts` (200k hot-path evals retain
  ~0 heap). Every stated boundary/limit is asserted in source and gathered in `tests/boundaries.test.ts`
  (REQ-ENG-004).

## Security practice (threat-informed; REQ-APP-172)

- **Threat-informed design.** Two red-team reviews: `spec/bsv-poker-spec-redteam-01.md` (spec) and
  `docs/redteam-02.md` (implementation — every finding re-adjudicated with code+test evidence).
- **Secure defaults.** Research/regtest by default; mainnet behind an explicit token and unmissable
  banner (`network-gate.ts`, REQ-PROD-012/REQ-VM-007); local services bind loopback-only
  (REQ-APP-106); the relay is untrusted and never interprets payloads (REQ-APP-070).
- **Key hygiene.** Keys live only inside the custody boundary (REQ-APP-101); logs/metrics/traces/
  diagnostic bundles never carry key material (`observability.ts` redaction, REQ-APP-124).
- **Trust-boundary validation.** Every inbound relay/peer envelope is validated before use; anything
  unrecognized is rejected (`message-validation.ts`, REQ-APP-103). Persisted records are validated on
  read and quarantined if corrupt (`persistence.ts`, REQ-APP-132).
- **No-fabrication invariants.** `OP_RETURN` banned in live scripts (lint + interpreter reject);
  negative tests fail inside the real Script interpreter; B1/B2 red-team regressions are guarded.
